  • frida如何实现native函数hook的？

hook 关键点就是找到函数地址

Frida 通过 Interceptor.attach 或 Interceptor.replace 实现 Native Hook：

Interceptor.attach(address, callbacks)：拦截 某个函数，执行自定义逻辑。

Interceptor.replace(address, new_function)：替换 某个函数，实现完全劫持。

在 Native 层，Frida 通过解析 ELF 文件，找到目标函数的地址，并在运行时修改 目标地址的指令，实现 Hook。

  • arm指令究竟是什么指令，能说说他与字节码指令的区别吗?

ARM 指令是 CPU 直接执行的低级指令，与硬件紧密相关，执行效率高。

字节码指令是 虚拟机指令集，需要解释或编译后才能执行，跨平台能力强但执行效率较低。

Android 应用最终运行的仍然是 ARM 指令，只是 Java 代码先转换为字节码，最后由 ART / JIT 编译成 ARM 机器码执行。

  • 汇编 函数参数传递的过程？
  1. 前 4 个整数参数（int, char \*, float 等）存放在寄存器 R0-R3 里。
  2. 第 5 个及以上的参数压入栈（Stack），从右向左依次入栈。
  3. 函数的返回值存放在 R0（或 R0-R1，如果返回 64 位数据）。
  4. 栈按 8 字节对齐（ARM EABI 规范）。
  5. LR（R14） 存储返回地址，调用 BL 指令时 LR 自动保存返回地址。
  • 汇编 函数返回有几种形式？
返回方式 描述 适用场景
BX LR 直接从 LR（Link Register）寄存器返回 最常见的返回方式
MOV PC, LR 通过 PC（程序计数器）恢复 LR 返回 与 BX LR 类似，但无条件返回
POP {PC} 从栈中恢复 PC，返回到调用者 适用于栈帧管理的函数
LDMFD SP!, {PC} 从栈中恢复 PC，返回到调用者 适用于多个寄存器恢复
  • r0~r15 寄存器的作用

参数寄存器：（R0-R3）

R7寄存器： 系统调用时，存放系统调用号，有时也用于作为FP使用。FP即栈基指针
R13寄存器：R13又名SP，即栈指针寄存器，指向当前程序栈顶，配合指令pop/push等。
R14寄存器：R14又名LR，即链接寄存器，存放函数的返回地址，和BL指令搭配；
R15寄存器：R15又名PC，即程序寄存器，存放CPU取指的地址

  • selinux安全相关

SELinux（Security-Enhanced Linux）是 Linux 内核 提供的一种 强制访问控制（MAC, Mandatory Access Control） 机制。

强制限制 App 访问系统资源，即使 App 获得 Root 权限，也不能随意修改系统文件。

防止恶意软件篡改系统进程（如 Hook system_server）。阻止 App 访问敏感文件（如 /data/system、/proc/kmsg）。

  • Android虚拟机中寄存器起什么作用，与栈的区别在哪里（又或者基于栈与基于寄存器的架构，谁更快？
  1. 寄存器架构（Register-Based）比栈架构（Stack-Based）快，因为它减少了指令数量，直接操作 CPU 寄存器，提高执行效率。
  2. JVM 采用栈架构，因为它适用于 跨平台环境，能在不同 CPU 结构上运行。
  3. Android 采用寄存器架构，因为它适用于 移动设备优化，减少功耗、提高性能。

  • Binder通信原理？

Binder 采用 C/S（Client/Server）架构，其通信流程如下：

  1. 服务端创建 Binder 对象（继承 Binder）。

  2. **服务端将 Binder 对象注册到 ServiceManager**。

  3. 服务端等待客户端调用。

  4. 客户端向 ServiceManager 查询 Binder。

  5. 获取 Binder 代理对象（Proxy）。

  6. 调用远程方法，Binder 驱动负责进程间通信。

  • 为什么可以删除section节区？

    ELF 的 Section 主要用于存储符号表、调试信息等，而程序运行时依赖的是 Program Header（程序头，PT_LOAD），ELF 运行时，加载器（Loader）只依赖 Program Header（程序头），而不依赖 Section Header。因此，可以删除 Section Headers 及其相关信息，而不影响程序的运行。

  • init_arry的作用，可以没有init_arry么？

init_array 是 ELF（Executable and Linkable Format） 文件中的一个特殊段（Section），用于存储程序在运行时自动执行的全局构造函数（Initializer Functions）。它的主要作用是在 main() 函数执行之前，初始化特定的函数或对象。

可以删除，但会影响全局构造函数的执行！init_array 主要用于全局对象构造函数，如果删除，C++ 全局对象、__attribute__((constructor)) 函数将不会执行。

  • 安卓启动后的第一个进程是什么，并说说你对他的理解

    init进程,它是一个由内核启动的用户级进程。内核自行启动(巳经被载入内存，开始运行,并已初始化所有的设备驱动程序和数据结构等)之后，系统就通过启动一个用户级程序init的方式来完成引导。init是第一个进程，进程的id为1。因为，系统的大部分服务都由它来启动，所以,init进程被赋予了很多极其重要的工作职责。 从它的代码（源码存在 system\core\init\init.c)来看，init进程的工作量还是很大的，主要集中在如下几个事情。 (1）解析 init.rc初始化脚本文件。 (2）初始化属性服务（property service )。 (3）进入无限 for循环，建立子进程，对关键服务的异常进行重启和异常处理。

  • 简述 Android 数字签名机制：

    Android 将数字签名用来标识应用程序的作者和应用程序之间建立的信任关系。这个数字证书并不需要权威的数字证书签名机构认证，它只是用来让应用程序包自我认证，也是用来判断该应用是否被别人破解，二次打包的一个标准，但是签名并不能防止被破解。

  • 数字签名有哪些优点？

    🔔 程序升级 只有相同的签名才认为是一个程序，包名也需要一样，才允许覆盖安装

    🔔 共享数据和代码 Android 提供了基于数字证书的权限赋予机制，应用程序可以和其他程序共享该功能或者将数据给哪些是与自己拥有相同数字证书的程序。如果应用在 manifest.xml 中声明了 permission 为 android:protectionLevel = “signature” , 则这个权限就只能授予那些根该权限所在包拥有同一数字签名的程序。

    🔔 模块化设计和开发 同一数字签名的程序运行在一个进程中，所以开发者可以将自己的程序分模块开发，而用户只需要在需要的时候下载合适的的模块

  1. 编译过程 2. 打包过程 3. 签名（不签名是无法安装的） 接下来详细说明三个部分的内容： 1. 编译过程 资源打包映射，处理AIDL文件，编译源代码（Java文件（分别为JVM和Dalvik）。 - 使用AAPT工具对资源文件进行打包，生成R.java文件 其中资源文件包括res目录下的布局、动画、图片、声音等文件,也包括AndroidManifest.xml文件。当然Android开发中还有一个目录是Asset目录，该目录下的资源不会被编译生成R处理，只是简单的压缩。 - 使用aidl工具处理AIDL文件,生成对应的.Java文件。 - 使用Javac命令编译Java文件,生成对应的.class文件。 - 把.class文件转化成 Davik VM(或ART VM）支持的.dex文件。 2. 打包过程 打包过程，即将上述步骤生成的文件，按照一定的格式压缩到一个文件，即APK文件中。其中主要的步骤是: 使用APK Builder打包生成未签名的APK文件。 打包后的资源文件、打包后dex文件、libs目录下的文件( so或第三方库文件)。 3. 签名优化过程 打包后的APK文件是不能够直接安装使用的，还需要对APK文件进行签名。签名优化过程是Android上特有的对APK安装包文件进行保护与优化的过程，没签名的APK文件系统是不允许安装的。其具体步骤是: - 使用jarsigner对未签名APK文件进行签名。（现在使用apksigner NDK 目录下） - 使用zipalign工具对签名后的APK文件进行对齐处理。 应用安装文件是随着Android的发展而发展来的，客户端的安装包安全也就需要尤为重视。了解了一个APK文件的生成过程，我们不仅仅可以使用一些批量脚本或工具完成APK生成，如Ant、Bash、Hudson工具等，我们还能够从逆向的角度分析一个 APK，了解如何重新打包APK,了解如何重新签名APK，这样才能了解如何保护好APK文件。

Dex-vmp原理：对原 dex 处理，找到要保护的方法，将原指令翻译成等价指令，加密存储，并将原指令替换为 VMP 入口指令 - 实现 VM，解释执行存储的等价指令。那么这样这样实现的VM可以在java层也可以在so层，一般来说是放在so层。

（1）解析dex： 定位保护的方法，可以手动解析dex文件或者使用开源库dexlib2

（2）将原指令翻译成等价指令 ： 这里就需要通过jni接口实现，这也是为什么dex-vmp加壳后有native标签，我们知道一条java指令对应可以用jni指令进行实现，那么我们需要的就是将对应的方法通过jni接口调用so层中c/c++实现方法进行翻译。因为dex-vmp需要调用jni接口，所以一定会调用CallVoidMethod方法等，所以这也是为什么很多快速追踪vmp调用栈选择InvokeWithVarArgs的原因

（3）实现VM: vm基于寄存器结构，我们需要初始化寄存器，然后对对应的Opcode进行翻译，这里调用一些jni接口

（4）解决JNI兼容的问题

arm-vmp原理：将so文件中的arm汇编的指令码转换为自定义的虚拟指令。核心是虚拟指令集和虚拟指令解释器

（1）将Native化的java代码和原C++代码使用Capstone进行反编译，生成原生汇编码

（2）使用随机生成的矩阵将原生指令码转换为虚拟指令码

（3）通过虚拟指令解释器对虚拟指令码进行解释

Unicorn 是一个 基于动态二进制翻译（DBT, Dynamic Binary Translation） 的 CPU 仿真器

可以简单理解为汇编代码的解释器，因此Unicorn执行汇编代码片段十分得心
应手。比如处理字符串加密（解密函数往往相对简单、独立，可以当作汇编代码片段），又比如处理花
指令（花指令确实就是汇编代码片段），除此之外，在去除指令替换等代码混淆上，也有用武之地。
